# Presentation-Oriented Brand Guide Writer Skill

## Instructions

- Quality is the priority above everything else. Speed does not matter. Do not call
  write_brand_guide until the result can closely reproduce the user's source material, not merely
  approximate its colors and logo.
- Hard gate: before write_brand_guide, create at least one local sample HTML slide/frame and render
  it to an image. If rendering tools are unavailable, explicitly report that constraint and do not
  pretend the quality loop happened.
- Hard gate: compare the rendered sample side-by-side against the source screenshot/page/deck.
  Revise and rerender until the sample matches the source's dominant background, composition,
  typography scale, spacing, asset fidelity, color balance, and signature effects.
- Hard gate: do not invent major visuals that are not in the source. For example, do not replace a
  source's black background and exact 3D/glass product object with a generic green sphere, planet,
  orbital graphic, abstract blob, or unrelated decorative shape.
- Analyze all user-provided source material first: URLs, screenshots, PDFs, PPTX files, images,
  existing decks, notes, and downloaded assets. Treat screenshots as visual ground truth.
- Extract everything useful for presentation design: exact assets, gradients, glows, shadows, 3D
  renders, textures, grids, line art, radii, borders, spacing systems, card styles, image
  treatments, typography scale, animation cues, composition patterns, and recurring section
  archetypes. When the source is a website or app, separate transferable brand signals from
  interface chrome. Do not carry over navigation bars, hamburger menus, search/login controls,
  carousel arrows, pause/play buttons, cookie banners, sticky headers, form controls, or other
  interactive UI as literal slide elements unless the user explicitly asks for a web UI kit. Use
  them only as evidence for slide-native qualities such as shape language, icon style, radius,
  spacing, contrast, motion cues, or CTA/callout treatment.
- If the source includes complex hero art or product visuals, download the exact image asset when
  possible. If it cannot be extracted exactly, describe how to recreate the effect with CSS/SVG in
  enough detail to match the screenshot.
- Generate one or more local sample slides or HTML frames against the extracted Brand Guide, compare
  them to the user's source URL/files/images, revise the Design Prompt, and repeat until the sample
  is recognizably on-brand. Do not stop after the first draft, and do not accept a sample with the
  wrong dominant background or invented hero art.
- Keep all scratch work in a dedicated temporary workspace such as
  `.onbrand/brand-guide-work/<brandGuideId>/`: downloaded source copies, extracted assets before
  upload, screenshots, sample HTML, rendered PNGs, notes, and comparison artifacts.
- Before the final response, clean up temporary brand-guide authoring artifacts unless the user
  explicitly asks to keep them. Never delete original user-provided files, uploaded S3 objects, or
  the persisted Brand Guide.
- Construct one complete Brand Guide containing one Brand Kit and one Presentation Kit.
- Extract exact logo/decorative asset files when available. Do not redraw, approximate, invent,
  trace, recolor, or replace logos. If an exact governed logo cannot be obtained, ask for it instead
  of fabricating one.
- Create role-aware Color Tokens with precise hex values and useful descriptions, not generic
  swatches. Use six-digit hex values; put opacity, gradient, and alpha usage in token descriptions
  and the Design Prompt.
- Write a detailed, presentation-oriented Design Prompt that is specific enough for a future agent
  to generate an on-brand HTML presentation without seeing the original source. The Design Prompt
  must translate source evidence into presentation-native systems: slide layouts, title hierarchy,
  section dividers, callout cards, chart treatments, agenda/progress markers, proof grids,
  framework diagrams, and closing CTA slides. It must not read like website recreation
  instructions.
- Call prepare_brand_guide_asset_uploads for exact asset files, upload them directly to S3 with the
  returned PUT commands, then call write_brand_guide with the returned assetId, s3Key, byteSize, and
  sha256 references. Asset IDs must be lowercase slugs such as 'primary-logo', 'hero-orb', or
  'icon-1'. Do not use consumer asset handles as upload IDs. Do not send asset bytes through MCP.

## Design Prompt Template

# NON-NEGOTIABLE ASSET HANDOFF

State every materialized asset handle and required local path. Tell the client never to redraw,
approximate, recolor, or replace exact governed assets.

---

# [Organization] Presentation Master Prompt (HTML/CSS)

## Source Evidence Inventory

List every source inspected and the visual evidence extracted from each: color values, typography,
logos, product/hero art, gradients, glows, shadows, textures, grids, 3D objects, shapes, cards,
spacing, screenshots, and section layouts. If the source is a website or app, explicitly classify
navigation, buttons, controls, menus, forms, carousel/pause controls, and other interactive chrome
as either transferable brand evidence or website-only UI. Call out anything that must be exact.

## Brand Essence

Capture the brand personality, market position, visual energy, and voice in concrete terms. Include
source-derived phrases and forbidden generic language.

## Visual Motifs and Effects

Describe the non-obvious visual language in implementation-ready detail: gradient recipes, glow
positions, blur radii, noise/grain, 3D or glass objects, orbital/grid systems, line art, shadows,
hover/animation cues, and how these motifs should translate into static slides.

## Website-to-Presentation Translation Rules

If any source evidence comes from a website or app, define how the brand translates to slides.
Separate transferable brand signals from website-only UI chrome. Do not reproduce navigation bars,
hamburger/search/login controls, carousel arrows, pause/play controls, cookie banners, sticky
headers, form fields, hover states, or other interactive affordances as literal slide elements unless
the user explicitly requested a web UI kit. Convert their visual qualities into presentation-native
components such as section labels, agenda markers, dividers, CTA/callout blocks, proof cards,
framework nodes, chart highlights, footers, or slide progress markers. Preserve typography, color,
spacing, motif, asset, and surface logic; discard interaction mechanics.

## Core CSS Tokens

Provide CSS custom properties for color tokens, typography stack, slide size, margins, gutters, and
recurring measurements.

## Typography

Specify headline, action title, section title, eyebrow, body, card title, data-label, and footnote
styles with sizes, weights, line heights, casing, and contrast rules.

## Layout Grammar

Define canvas, grids, spacing, composition patterns, panel geometry, border/radius/shadow rules,
density, alignment, and what not to use. Include numeric placement guidance for covers and key slide
archetypes.

## Asset Usage

For each Logo and Decorative Asset, specify local path, size ranges, placement, clear space, allowed
contexts, disallowed transformations, and layering behavior.

## Required Slide Types

Define cover, executive summary, problem/opportunity, solution/framework, proof/case study, roadmap,
metrics/exhibit, and closing slides using brand-specific layouts.

## Chart and Data Rules

Specify chart types, palettes, gridlines, labels, source lines, emphasis colors, and complexity
limits.

## Copy Rules

Define action-title behavior, vocabulary, sentence style, tone, and terms to avoid.

## Sample Generation and Self-Critique

Before writing the Brand Guide, generate representative sample slides/HTML frames for the most
visually demanding source sections, then render them to images. Compare the rendered samples against
the user's screenshots/files/URL side-by-side. Revise the Design Prompt, regenerate, and rerender
until the samples preserve the same dominant background, composition, spacing, typography, color
balance, effects, asset fidelity, and brand feel. Record what changed between iterations and the
final pass/fail criteria here. If you cannot render locally, say so explicitly and do not claim
visual validation.

Reject samples that are merely adjacent to the brand. A sample fails if it uses the wrong background
value, replaces source hero/product art with generic invented shapes, changes the source's spatial
hierarchy, loses signature gradients/glows/textures, or feels like a generic themed deck rather than
the supplied material.

## Temporary Workspace and Cleanup

During authoring, keep downloaded source copies, extracted asset candidates, screenshots, sample
HTML, rendered PNGs, comparison notes, and other scratch artifacts in a dedicated temporary
workspace such as .onbrand/brand-guide-work/[brandGuideId]/. After the Brand Guide is successfully
written, delete that temporary workspace unless the user explicitly asks to keep it. Never delete
original user-provided files, uploaded S3 objects, or the persisted Brand Guide.

## Quality Checklist

List concrete pass/fail checks for assets, palette, typography, layout, contrast, storyline,
dependency constraints, and visual fidelity to the supplied source material. Include explicit checks
for dominant background color, source-faithful hero/product imagery, no invented major decorative
objects, screenshot-level composition, and whether rendered sample validation was completed.

## Quality Bar

A Brand Guide is not acceptable just because it has the right logo and colors. It must capture the
distinctive page/deck mechanics that make the source recognizable, and it must survive rendered
sample validation before persistence.

### Extraction Checklist

- Exact logo and all reusable visual assets.
- Hero/product imagery, including 3D objects, glass effects, glow fields, gradients, and shadows.
- Background systems such as grids, dots, orbital lines, noise, textures, and section dividers.
- Website/app UI evidence: navigation, buttons, chips, accordions, forms, controls, cards, tables,
  and customer/proof modules. Classify each as transferable brand signal or website-only chrome;
  translate transferable qualities into slide-native components rather than literal webpage UI.
- Typography family, weight, optical feel, scale, line height, tracking, and hierarchy.
- Spacing and composition: margins, gutters, content widths, asymmetry, density, and focal points.
- Animation or interaction cues translated into static slide rules.
- Concrete presentation do/don't rules derived from source screenshots, not generic SaaS defaults
  and not literal webpage recreation.
- Dominant background colors and surface hierarchy measured from rendered source material, not
  guessed from memory.
- Explicit list of forbidden invented visuals that would degrade fidelity, such as generic planets,
  spheres, blobs, or decorative objects absent from the source.

### Self-Critique Loop

- Build representative sample slides or HTML frames locally before calling write_brand_guide.
- Render those samples to images; visual validation requires rendered pixels, not prose-only
  confidence.
- Compare samples against the user's source material side-by-side.
- Identify mismatches in composition, proportions, dominant background, visual effects, asset
  fidelity, color balance, and typography.
- Revise the Design Prompt and samples, rerender, and repeat until the result is recognizably
  aligned with the source.
- Only then persist the Brand Guide. If the loop cannot be run, disclose that instead of silently
  skipping it.
- Clean up temporary authoring artifacts after persistence unless the user asked to keep them.
