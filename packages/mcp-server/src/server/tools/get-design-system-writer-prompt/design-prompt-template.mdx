# [Organization] Presentation Master Prompt (HTML/CSS)

## Source Evidence Inventory

List every source inspected and the visual evidence extracted from each: color values, typography,
logos, product/hero art, gradients, glows, shadows, textures, grids, 3D objects, shapes, cards,
navigation, buttons, spacing, screenshots, and section layouts. Call out anything that must be
exact.

## Brand Essence

Capture the brand personality, market position, visual energy, and voice in concrete terms. Include
source-derived phrases and forbidden generic language.

## Visual Motifs and Effects

Describe the non-obvious visual language in implementation-ready detail: gradient recipes, glow
positions, blur radii, noise/grain, 3D or glass objects, orbital/grid systems, line art, shadows,
hover/animation cues, and how these motifs should translate into static slides.

## Core CSS Tokens

Provide CSS custom properties for color tokens, typography stack, slide size, margins, gutters, and
recurring measurements.

## Typography

Specify headline, action title, section title, eyebrow, body, card title, data-label, and footnote
styles with sizes, weights, line heights, casing, and contrast rules.

## Layout Grammar

Define canvas, grids, spacing, composition patterns, panel geometry, border/radius/shadow rules,
density, alignment, and what not to use. Include numeric placement guidance for covers and key slide
archetypes.

## Asset Usage

For each Logo and Decorative Asset, specify local path, size ranges, placement, clear space, allowed
contexts, disallowed transformations, and layering behavior.

## Required Slide Types

Define cover, executive summary, problem/opportunity, solution/framework, proof/case study, roadmap,
metrics/exhibit, and closing slides using brand-specific layouts.

## Chart and Data Rules

Specify chart types, palettes, gridlines, labels, source lines, emphasis colors, and complexity
limits.

## Copy Rules

Define action-title behavior, vocabulary, sentence style, tone, and terms to avoid.

## Sample Generation and Self-Critique

Before writing the Design System, generate representative sample slides/HTML frames for the most
visually demanding source sections, then render them to images. Compare the rendered samples against
the user's screenshots/files/URL side-by-side. Revise the Design Prompt, regenerate, and rerender
until the samples preserve the same dominant background, composition, spacing, typography, color
balance, effects, asset fidelity, and brand feel. Record what changed between iterations and the
final pass/fail criteria here. If you cannot render locally, say so explicitly and do not claim
visual validation.

Reject samples that are merely adjacent to the brand. A sample fails if it uses the wrong background
value, replaces source hero/product art with generic invented shapes, changes the source's spatial
hierarchy, loses signature gradients/glows/textures, or feels like a generic themed deck rather than
the supplied material.

## Temporary Workspace and Cleanup

During authoring, keep downloaded source copies, extracted asset candidates, screenshots, sample
HTML, rendered PNGs, comparison notes, and other scratch artifacts in a dedicated temporary
workspace such as .onbrand/design-system-work/[designSystemId]/. After the Design System is
successfully written, delete that temporary workspace unless the user explicitly asks to keep it.
Never delete original user-provided files, uploaded S3 objects, or the persisted Design System.

## Quality Checklist

List concrete pass/fail checks for assets, palette, typography, layout, contrast, storyline,
dependency constraints, and visual fidelity to the supplied source material. Include explicit checks
for dominant background color, source-faithful hero/product imagery, no invented major decorative
objects, screenshot-level composition, and whether rendered sample validation was completed.
